## [cite_start]บทที่ 1 บทนำ [cite: 51]

### [cite_start]1.1 ที่มาและความสำคัญของปัญหา (Background and Problem Statement) [cite: 52]
[cite_start]ธุรกิจร้านขายยาเป็นธุรกิจที่มีความละเอียดอ่อน เนื่องจากเกี่ยวข้องกับสุขภาพและความปลอดภัยของผู้บริโภค สินค้าภายในร้านมีจำนวนรายการ (SKU) ที่หลากหลาย ทั้งยาสามัญ ยาอันตราย เวชภัณฑ์ และอาหารเสริม ซึ่งสินค้าแต่ละชนิดมีเงื่อนไขการจัดเก็บที่แตกต่างกัน โดยเฉพาะอย่างยิ่งเงื่อนไขเรื่อง "วันหมดอายุ (Expiration Date)" และ "หมายเลขการผลิต (Lot/Batch Number)" [cite: 53]

[cite_start]จากการศึกษาปัญหาในปัจจุบัน พบว่าร้านยาที่ยังไม่มีระบบจัดการฐานข้อมูล หรือใช้การจดบันทึกด้วยมือ (Manual System) มักประสบปัญหาสำคัญดังนี้: [cite: 54]
1. [cite_start]ปัญหาด้านการบริหารสต็อกและวันหมดอายุ: ไม่สามารถตรวจสอบได้ว่ายาตัวใดใกล้หมดอายุ ทำให้ไม่สามารถระบายสินค้าได้ทันตามหลัก FEFO (First Expire, First Out) ส่งผลให้เกิดความเสียหายทางต้นทุน [cite: 55]
2. [cite_start]ความผิดพลาดของข้อมูล: ข้อมูลสินค้าคงคลังจริงไม่ตรงกับบัญชี (Stock Discrepancy) เนื่องจากการตัดสต็อกไม่เป็น Real-time [cite: 56]
3. [cite_start]ความล่าช้าในการสืบค้นข้อมูล: การค้นหาประวัติการซื้อยาของลูกค้า หรือการค้นหาราคาทุนเพื่อคำนวณกำไรทำได้ยากและใช้เวลานาน [cite: 57]
4. [cite_start]ความซ้ำซ้อนของข้อมูล: ข้อมูลมีการจัดเก็บซ้ำซ้อน ไม่เป็นระเบียบ ขัดต่อหลักการ Normalization [cite: 58]

[cite_start]ดังนั้น คณะผู้จัดทำจึงเล็งเห็นความสำคัญในการนำระบบฐานข้อมูลเชิงสัมพันธ์ (Relational Database System) มาประยุกต์ใช้เพื่อพัฒนาระบบบริหารจัดการร้านขายยา เพื่อให้การดำเนินงานเป็นไปอย่างมีประสิทธิภาพ ลดความผิดพลาด และสนับสนุนการตัดสินใจทางธุรกิจ [cite: 59]

### [cite_start]1.2 วัตถุประสงค์ของโครงงาน (Objectives) [cite: 60]
1. [cite_start]เพื่อวิเคราะห์และออกแบบฐานข้อมูล (Database Design) ที่ถูกต้องตามหลักการ Normalization เพื่อลดความซ้ำซ้อนของข้อมูล [cite: 61]
2. [cite_start]เพื่อพัฒนาระบบฐานข้อมูลที่รองรับการจัดการสินค้าแบบหลายลอต (Multi-batch Inventory) สำหรับควบคุมวันหมดอายุ [cite: 62]
3. [cite_start]เพื่อสร้างระบบบันทึกการขายและตัดสต็อกสินค้า (Point of Sale) ที่มีความถูกต้องแม่นยำ [cite: 63]
4. [cite_start]เพื่อประยุกต์ใช้ภาษา SQL ในการจัดการข้อมูล (Data Manipulation) และการดึงข้อมูลเพื่อทำรายงาน (Reporting) [cite: 64]

### [cite_start]1.3 ขอบเขตของโครงงาน (Scope of Project) [cite: 65]
[cite_start]**1.3.1 ขอบเขตด้านผู้ใช้งาน (User Roles)** [cite: 66]
* [cite_start]Administrator (เจ้าของร้าน): มีสิทธิ์เข้าถึงข้อมูลทุกส่วน เพิ่ม/ลบ/แก้ไข ข้อมูลหลัก (Master Data), อนุมัติการสั่งซื้อ, และดูรายงานสรุปยอดขาย [cite: 67]
* [cite_start]Pharmacist (เภสัชกร/พนักงานขาย): มีสิทธิ์ใช้งานระบบขายหน้าร้าน (POS), ค้นหาสินค้า, ตรวจสอบวันหมดอายุ, และสมัครสมาชิกให้ลูกค้า [cite: 68]

[cite_start]**1.3.2 ขอบเขตด้านระบบงาน (Functional Scope)** [cite: 69]
* [cite_start]ระบบจัดการข้อมูลหลัก (Master Data Management): จัดการข้อมูลยา, ประเภทสินค้า, ข้อมูลซัพพลายเออร์, ข้อมูลพนักงาน [cite: 70]
* [cite_start]ระบบบริหารคลังสินค้า (Inventory Management): รับสินค้าเข้าคลังโดยระบุ Lot และวันหมดอายุ, ตรวจสอบสินค้าคงเหลือ [cite: 71]
* [cite_start]ระบบขายหน้าร้าน (Sale & POS): คำนวณยอดเงิน, ตัดสต็อกตาม Lot (FEFO), ออกใบเสร็จรับเงิน [cite: 72]
* [cite_start]ระบบรายงาน (Reporting): รายงานยอดขายรายวัน/เดือน, รายงานสินค้าใกล้หมดอายุ, รายงานสินค้าขายดี [cite: 73]
* [cite_start]ระบบจัดซื้อสินค้า (Purchase Management): รองรับกระบวนการสั่งซื้อสินค้าจากคู่ค้า (Ordering) โดยสามารถสร้างใบสั่งซื้อ (Purchase Order) บันทึกรายละเอียดสินค้าที่สั่งซื้อ, คำนวณต้นทุนรวม และระบุพนักงานผู้รับผิดชอบได้ [cite: 74] [cite_start]เมื่อมีการรับสินค้าเข้าจริง ระบบจะทำการสร้างข้อมูลลอตสินค้า (Product_Batches) อัตโนมัติ โดยผูกข้อมูลกับใบสั่งซื้อและคู่ค้าที่เกี่ยวข้อง เพื่อรองรับการตรวจสอบย้อนกลับ [cite: 75, 76]

### [cite_start]1.4 ประโยชน์ที่คาดว่าจะได้รับ (Expected Benefits) [cite: 77]
1. [cite_start]ได้ต้นแบบฐานข้อมูลร้านยาที่ถูกต้องตามหลักวิชาการ สามารถนำไปพัฒนาต่อเป็นซอฟต์แวร์จริงได้ [cite: 78]
2. [cite_start]ลดความเสียหายจากยาหมดอายุคาสต็อก ด้วยการบริหารจัดการแบบ Lot Control [cite: 79]
3. [cite_start]ทราบต้นทุนและกำไรที่แท้จริงจากการขายในแต่ละช่วงเวลา [cite: 80]
4. [cite_start]เพิ่มความสะดวกรวดเร็วในการให้บริการลูกค้า [cite: 81]

## [cite_start]บทที่ 2 ทฤษฎีและเทคโนโลยีที่เกี่ยวข้อง [cite: 82, 83]

[cite_start]**2.1 ระบบฐานข้อมูลเชิงสัมพันธ์ (Relational Database)** [cite: 85]
[cite_start]ศึกษาโครงสร้างการจัดเก็บข้อมูลในรูปแบบตาราง (Table) การกำหนดคีย์หลัก (Primary Key) คีย์นอก (Foreign Key) และความสัมพันธ์ระหว่างตาราง (Relationships) แบบ One-to-One, One-to-Many และ Many-to-Many [cite: 86, 87]

[cite_start]**2.2 การนอร์มัลไลเซชัน (Normalization)** [cite: 88]
[cite_start]ประยุกต์ใช้กฎ Normalization เพื่อลดความซ้ำซ้อนของข้อมูลและป้องกันความผิดปกติของข้อมูล (Data Anomaly) โดยจะทำการจัดระเบียบข้อมูลให้อยู่ในรูปแบบ: [cite: 89, 90]
* [cite_start]First Normal Form (1NF): กำจัดกลุ่มข้อมูลที่ซ้ำกัน [cite: 91]
* [cite_start]Second Normal Form (2NF): กำจัดการขึ้นต่อกันบางส่วน (Partial Dependency) [cite: 92]
* [cite_start]Third Normal Form (3NF): กำจัดการขึ้นต่อกันแบบทรานซิทีฟ (Transitive Dependency) [cite: 93]

[cite_start]**2.3 เครื่องมือที่ใช้ในการพัฒนา (Tools)** [cite: 94]
* [cite_start]Language & Web: HTML5, CSS3, JavaScript [cite: 95]
* [cite_start]Server Runtime: Node.js (Express) [cite: 96]
* [cite_start]Database: Oracle DB, MySQL [cite: 97]
* [cite_start]Tools & Client: MySQL Workbench, oracledb connector [cite: 98]
* [cite_start]DevOps: Docker [cite: 99]

## [cite_start]บทที่ 3 การวิเคราะห์และออกแบบระบบ [cite: 101, 102]

### [cite_start]3.1 กฎเกณฑ์การดำเนินธุรกิจ (Business Rules) [cite: 103]
1. [cite_start]**สินค้า 1 ชนิด สามารถมีได้หลายลอตการผลิต (One Product has many Batches)** ในร้านยา ยาหนึ่งชนิดจะมีการรับเข้ามาหลายครั้ง แต่ละครั้งจะมี "วันหมดอายุ" และ "เลขล็อต" ที่ต่างกัน [cite: 105, 106]
2. [cite_start]**หมวดหมู่ 1 หมวด สามารถมีสินค้าได้หลายชิ้น (One Product belongs to one Category)** เป็นการจัดระเบียบสินค้าให้เป็นหมวดหมู่ชัดเจน เช่น ยาแก้ปวด, ยาปฏิชีวนะ [cite: 107, 108]
3. [cite_start]**ใบเสร็จ 1 ใบ สามารถมีรายการสินค้าได้หลายรายการ (One Sales Order has many Sales Details)** ลูกค้าสามารถซื้อยาได้หลายอย่างพร้อมกัน [cite: 109, 110]
4. [cite_start]**คู่ค้า 1 ราย สามารถจัดส่งลอตสินค้าได้หลายลอต (One Supplier supplies many Batches)** ต้องระบุว่ามาจากคู่ค้าเจ้าไหน (Supplier) เพื่อตรวจสอบย้อนกลับได้ [cite: 111, 112]
5. [cite_start]**ลอตสินค้า 1 ลอต สามารถอยู่ในรายละเอียดสินค้าได้หลายรายการ (Each Sales Detail refers to one Batch)** เวลาหยิบสินค้าขายต้องระบุว่าหยิบจากลอตไหน เพื่อให้ระบบไปตัดยอดคงเหลือจากลอตนั้นๆ ได้ถูกต้องตามความเป็นจริง [cite: 113, 114]
6. [cite_start]**พนักงาน 1 คน สามารถทำรายการขายได้หลายใบเสร็จ (One Employee processes many Sales)** เพื่อระบุความรับผิดชอบว่าใครเป็นคนบันทึกยอดขายและรับเงินในบิลนั้นๆ [cite: 115, 116]

### [cite_start]3.5 Workflow (กระบวนการทำงาน) [cite: 290]

**1. [cite_start]กระบวนการรับสินค้าเข้าสต็อก (Receiving & Stock In)** [cite: 291]
[cite_start]เป้าหมาย: เพื่อบันทึกสินค้าใหม่ที่รับมาจากคู่ค้า ลงในตารางลอตสินค้า (Product_Batches) [cite: 292]
* เลือกคู่ค้า (Identify Supplier): เลือกรายชื่อคู่ค้าจากระบบ ถ้าเป็นคู่ค้าใหม่ทำการเพิ่มข้อมูลลงตาราง Suppliers ก่อน [cite: 294, 295, 296]
* [cite_start]ระบุตัวสินค้า (Identify Product): ค้นหาสินค้าที่รับเข้า ถ้าเป็นสินค้าใหม่ทำการสร้างรหัสสินค้าลงตาราง Products [cite: 297, 298, 299]
* [cite_start]บันทึกข้อมูลลอต (Batch Entry): กรอกข้อมูล Lot Number, MFG_Date, EXP_Date, Cost_Price, จำนวนรับเข้า [cite: 301, 302, 303, 304, 305, 306]
* [cite_start]ตรวจสอบความซ้ำซ้อน: ระบบตรวจสอบว่า Product_ID คู่กับ Lot_Number และ Supplier รายนี้มีอยู่ในระบบแล้วหรือไม่ ถ้ามีให้แจ้งเตือนเพิ่มจำนวนทบ [cite: 307, 308, 309]
* ยืนยันการรับเข้า (Commit): ระบบบันทึกข้อมูลลงตาราง Product_Batches โดยผูก Supplier_ID เข้ากับลอตสินค้านั้นๆ ทันที [cite: 310, 311, 312]

**2. [cite_start]กระบวนการขายหน้าร้าน (Point of Sale - POS)** [cite: 313]
[cite_start]เป้าหมาย: เพื่อขายสินค้า ออกใบเสร็จ และตัดสต็อกสินค้าตามลอตที่ถูกต้อง (FEFO) [cite: 314]
* [cite_start]เปิดบิลใหม่ (Initialize Sale): ระบบสร้างเลขที่ใบเสร็จ (Sale_ID) รอไว้ในหน่วยความจำ บันทึกวันที่และพนักงานที่ล็อกอินอยู่ [cite: 319, 320, 323]
* [cite_start]สแกน/เลือกสินค้า: พนักงานยิงบาร์โค้ด [cite: 324, 325]
* ระบบเลือกตัดสต็อกอัตโนมัติ (System Auto-Batch Selection): ค้นหาตาราง Product_Batches ใช้ลอจิก FEFO เรียงลำดับลอตที่ EXP_Date หมดอายุเร็วที่สุด และยังมี Quantity_Remaining > 0 [cite: 326, 327, 328, 329, 330]
* [cite_start]ระบุจำนวนขาย: ถ้ายอดซื้อมากกว่าจำนวนที่เหลือในลอตแรก ระบบจะไปดึง Batch_ID ของลอตถัดไปมาเพิ่มอัตโนมัติ (Split Line Item) [cite: 331, 332, 333]
* ปิดบิล (Payment & Finalize): ระบบเริ่ม Transaction, Insert Sales_Header, Insert Sales_Detail, Update Product_Batches (ตัดสต็อก), และ Commit [cite: 336, 338, 339, 340, 341, 342, 343]

**3. [cite_start]การตรวจสอบสต็อกและสั่งซื้อ (Inventory & Reordering)** [cite: 346]
* ตรวจสอบจุดสั่งซื้อ (Check Reorder Point): ระบบรวมจำนวนคงเหลือ (SUM(Quantity_Remaining)) เปรียบเทียบกับค่า Reorder_Point ถ้าต่ำกว่าระบบแจ้งเตือน [cite: 349, 350, 352, 353]
* [cite_start]ตรวจสอบสินค้าใกล้หมดอายุ (Check Expiry): กรองเฉพาะลอตที่ใกล้หมดอายุ เพื่อจัดโปรโมชั่น หรือส่งคืน [cite: 354, 356, 358]

FOR EACH ROW
BEGIN
    IF INSERTING THEN
        INSERT INTO Product_Audit_Log (Action_Type, Product_ID, Product_Name, New_Price, Details)
        VALUES ('INSERT', :NEW.Product_ID, :NEW.Product_Name, :NEW.Unit_Price, 'เพิ่มสินค้าใหม่');
    ELSIF UPDATING THEN

    END IF;
END;

    p_emp_id IN VARCHAR2,
    p_username IN VARCHAR2,
    p_password IN VARCHAR2,
    p_position IN VARCHAR2,
    p_first_name IN VARCHAR2,
    p_last_name IN VARCHAR2,
    p_action IN VARCHAR2,
    p_result OUT VARCHAR2
) AS
BEGIN
    IF UPPER(p_action) = 'INSERT' THEN
        INSERT INTO Employees (EMP_ID, Username, Password_Hash, Position, First_Name, Last_Name)
        VALUES (p_emp_id, p_username, p_password, p_position, p_first_name, p_last_name);
        p_result := 'SUCCESS: เพิ่มพนักงานสำเร็จ';
    ELSIF UPPER(p_action) = 'UPDATE' THEN
        IF p_password IS NOT NULL AND LENGTH(TRIM(p_password)) > 0 THEN
            UPDATE Employees
            SET Username = p_username, Password_Hash = p_password, Position = p_position,
                First_Name = p_first_name, Last_Name = p_last_name
            WHERE TRIM(EMP_ID) = TRIM(p_emp_id);
        ELSE
            UPDATE Employees
            SET Username = p_username, Position = p_position,
                First_Name = p_first_name, Last_Name = p_last_name
            WHERE TRIM(EMP_ID) = TRIM(p_emp_id);
        END IF;
        p_result := 'SUCCESS: แก้ไขพนักงานสำเร็จ';
    ELSE
        p_result := 'ERROR: action ต้องเป็น INSERT หรือ UPDATE';
    END IF;
EXCEPTION
    WHEN DUP_VAL_ON_INDEX THEN
        p_result := 'ERROR: รหัสพนักงานซ้ำ';
    WHEN OTHERS THEN
        p_result := 'ERROR: ' || SQLERRM;
END;
